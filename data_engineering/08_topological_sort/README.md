💻 【外資IT企業（DE/FDE職）定番過去問】
課題：dbtデータ変換モデルのコンパイルエンジン（DAGトポロジカルソート）
【ビジネス背景・お題】
アナリストが作成した多数のdbt（data build tool）のデータ変換モデルと、それらの「依存関係（どのテーブルを先に作らないといけないか）」のメタデータがあります。

あんたはデータエンジニアとして、これらのモデルをデータウェアハウス（Snowflakeなど）に対して「依存関係が破綻しないように、正しい順番で実行（ビルド）するコンポーネント」をスクラッチで実装する必要があります。

また、アナリストがバグで「循環参照（無限ループ）」を設定してしまった場合、システムが無限に回り続けてインフラ破産するのを防ぐため、即座にそれを検知してエラー（例外）を投げる堅牢さも求められます。

📌 満たすべきビジネスルールと要件：

トポロジカルソート（Topological Sort）の実装：
すべてのモデル（テーブル）を、依存している親テーブルが「必ず先にビルドされる」ような正しい順番に並び替えた配列（リスト）を返してください。

循環参照（Circular Dependency）の検知：
もし、テーブルA が テーブルB に依存し、テーブルB が テーブルC に依存し、テーブルC が テーブルA に依存しているような「無限ループ（循環）」を発見した場合、処理を中断して ValueError を発生させなさい（Fail-Fast思想ね）。

入力の独立性（孤立モデルの許容）：
どこにも依存していないし、誰からも依存されていない完全に孤立したテーブル（例：単なるマスタテーブルのロードなど）があっても、システムを落とさずに任意の順番（最初など）に含めて処理しなさい。

📥 入力データの仕様
データの依存関係は、以下のような「ハッシュマップ（辞書）」の形式で渡されます。
キーが「作成したいモデル名」、値が「事前に作成が必須なモデル名のリスト（親たち）」よ。

```python

# 入力データのイメージ
dependency_graph = {
    "stg_users": [],                    # 親はなし（Rawデータから直接作る）
    "stg_orders": [],                   # 親はなし
    "dim_users": ["stg_users"],         # stg_users が先に作られてないとダメ
    "fct_orders": ["stg_orders", "dim_users"], # 2つのテーブルに依存している
    "analytics_dashboard": ["fct_orders"] # 最下流のモデル
}

```

📤 期待される出力（返却値）
正しくビルドできる順番に並べ替えた、モデル名のリスト List[str] を返しなさい。
上の入力データの場合、期待される出力の1つの正解はこうなるわ：

```python

["stg_users", "stg_orders", "dim_users", "fct_orders", "analytics_dashboard"]

```

🛠️ 挑戦のためのテンプレートコード深さ優先探索（DFS）と、各ノードの「探索状態」をハッシュマップ（または配列）で管理するテクニック（3色判定法などと呼ばれるわ）を使うのが、最もスマートで計算量 $O(V + E)$ に抑えられるシニアの書き方よ。まずはこの構造から、カンペなしで組んでみて！
