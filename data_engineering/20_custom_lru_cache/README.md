
課題20：LRUキャッシュ（Least Recently Used Cache）のスクラッチ実装

### 【ユースケースとシステム背景】
あんたはリアルタイムデータ処理基盤のシニアエンジニアです。
高頻度でアクセスされるデータ（例: ユーザープロフィール、APIレスポンス、MLモデルの推論結果）をメモリ上にキャッシュして、バックエンドDBへのアクセスを最小限に抑える必要があります。

しかし、メモリは有限です。キャッシュの容量（capacity）が上限に達したら、**「一番長い間使われていないデータ」を自動的に追い出す（evict）** 仕組みが必要です。これが LRU（Least Recently Used）キャッシュです。

Pythonの `collections.OrderedDict` を使わず、**ハッシュマップ（辞書）＋ 双方向連結リスト（Doubly Linked List）を自作**して実装してください。

---

### 📌 満たすべきインターフェースと要件：

#### `LRUCache(capacity: int)`
- 指定された `capacity`（正の整数）でキャッシュを初期化する。

#### `get(key: int) -> int`
- `key` に紐づく値を返す。存在しない場合は `-1` を返す。
- アクセスされたデータは「最近使われた」として、内部の順序を更新する。

#### `put(key: int, value: int) -> None`
- `key` に対して `value` を格納する。
- すでにキーが存在する場合は値を上書きし、「最近使われた」として順序を更新する。
- キャッシュが `capacity` を超える場合は、**最も長い間使われていないデータ（LRU）を1つ追い出してから**、新しいデータを格納する。

---

### 📌 制約と要件（シニアの防衛策）：

1. **計算量の最適化**:
   - `get` と `put` の両方が **O(1)** の時間計算量で動作すること。
   - O(N) の線形探索（リストの走査）を使った実装は不合格とします。
2. **データ構造の自作**:
   - `collections.OrderedDict` を使わず、`dict` + 双方向連結リスト（自作のNodeクラス）で実装すること。
   - なぜ「双方向」である必要があるのか（O(1)でのノード削除のため）を理解しておくこと。
3. **エッジケースの処理**:
   - `capacity = 1` の場合でも正しく動作すること。
   - 存在しないキーを `get` しても `-1` を返し、クラッシュしないこと。

---

### 📥 入力と期待される動作のイメージ

```python
cache = LRUCache(capacity=2)

cache.put(1, 100)      # キャッシュ: {1: 100}
cache.put(2, 200)      # キャッシュ: {1: 100, 2: 200}
cache.get(1)           # -> 100 (キー1にアクセス -> 1が「最近使われた」に昇格)
cache.put(3, 300)      # 容量オーバー! キー2がLRU（最も古い）なので追い出し
                       # キャッシュ: {1: 100, 3: 300}
cache.get(2)           # -> -1 (キー2は追い出し済み)
cache.put(4, 400)      # 容量オーバー! キー1がLRUなので追い出し
                       # キャッシュ: {3: 300, 4: 400}
cache.get(1)           # -> -1 (キー1は追い出し済み)
cache.get(3)           # -> 300
cache.get(4)           # -> 400
```

---

### 💡 アルゴリズムのヒント: 「HashMap + Doubly Linked List」パターン

**なぜこの2つの組み合わせが必要なのか？**

1. **辞書（HashMap）だけではダメな理由**:
   辞書は O(1) でキーから値を引けるが、「どれが一番古いか（使用順序）」を管理できない。

2. **リスト（配列）だけではダメな理由**:
   リストは順序を管理できるが、「特定のキーのノードを探して削除する」のに O(N) かかる。

3. **組み合わせの強み**:
   - 辞書: キーから O(1) でノードの「場所」を引く。
   - 双方向連結リスト: ノードを O(1) で削除（前後のポインタを繋ぎ替えるだけ）し、先頭に O(1) で移動させる。

### 💡 本実装パターンの重要性と実務上の価値

- **データ構造の深い理解**: 配列・辞書・連結リストの長所短所を理解し、組み合わせて O(1) を実現する設計力。
- **ポインタ操作の正確性**: 双方向連結リストのノード追加・削除は、バグが生まれやすい箇所。正確にポインタを繋ぎ替えられるかが試される。
- **DE実務への直結**: Redis, Memcached, CDNキャッシュなど、あらゆるデータ基盤の裏側で使われている仕組みの理解を証明できる。
