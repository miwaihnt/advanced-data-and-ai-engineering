# 課題14：マルチプロセス並列処理による巨大JSONLの高速パース（Bypassing GIL with ProcessPoolExecutor）

### 【ビジネス背景・お題】
あなたは巨大な決済プラットフォームのデータインフラを運用しています。
毎日、数千万件におよぶ大量の「トランザクションイベントログ（JSONL形式）」が生成され、これを解析してユーザーごとの「合計取引金額」を集計する日次バッチ処理を書いています。

ログファイルの容量は数十ギガバイトに及び、単純なシングルスレッド処理では数時間かかってしまいます。
さらに、Pythonには**GIL（グローバルインタプリタロック）**が存在するため、マルチスレッド（`Threading`）を使ってもCPUコアを1つしか使えず、JSONのデコードというCPUバウンドな処理を高速化することはできません。

マルチプロセス並列処理モジュール `concurrent.futures.ProcessPoolExecutor` を駆使し、マシンの全CPUコアを限界まで使い倒して、OOM（メモリ不足）を起こさずに超高速でログをパース・集計する堅牢なパイプラインを実装しなさい。

---

### 🚨 面接官がチェックする極限状態の要件：

1. **GILの完全回避とマルチコア利用:**
   `ProcessPoolExecutor` を用いて、CPUバウンドなJSONパース処理を複数の独立したプロセスに分散させ、マルチコアCPUの恩恵を最大化しなさい。

2. **メモリ効率（ジェネレータによるチャンキング）:**
   数GB〜数編GBのログファイルを一度にメモリにロードするのはOOM（メモリ枯渇）で即死するため絶対に不可です。
   ファイルを一定行数（例: 1,000行）ずつの「チャンク（塊）」として読み込むジェネレータを作成し、それをワーカープロセスに順次ディスパッチしなさい（空間計算量 $O(ChunkSize \times CPU数)$）。

3. **MapReduce（マッピングと集約）の設計:**
   - **Mapフェーズ (各プロセス内)**: 割り当てられたテキスト行のリスト（チャンク）を受け取り、各行をJSONパースし、有効なトランザクションのみから「ローカルでの集計辞書 `{user_id: total_amount}`」を作成して返します。
   - **Reduceフェーズ (メインプロセス内)**: 各プロセスから返ってきた部分集計結果を受け取り、メインプロセス上のマスター辞書にマージします。

4. **例外耐性と耐久性:**
   ログデータの中には、一部壊れたJSON文字列が含まれることがあります。1行のパースエラーでプロセス全体をクラッシュさせず、警告をログ出力して処理を継続させなさい。

---

### 📥 入力データの仕様

入力は JSONL 形式のテキストファイルです。
各行は以下のスキーマを持つ JSON 形式になっています。
```json
{"user_id": "u1", "amount": 5000, "status": "success"}
{"user_id": "u2", "amount": 1200, "status": "failed"}  // statusがsuccess以外のものは集計対象外
```

### 💡 この課題で学べること（シニアDEの必須スキル）

- **GIL（Global Interpreter Lock）の物理限界の理解:** PythonがCPUバウンド処理でスレッドではなくプロセスを必要とする理由。
- **IPC（プロセス間通信）のコスト:** `multiprocessing` において、生データを丸ごと渡すのではなく、部分集計した小さな辞書だけを返すことでシリアライズ/デシリアライズの通信オーバーヘッドを劇的に減らす設計手法。
- **ファイルI/Oのブロッキング回避:** ジェネレータを用いたスマートなチャンキング。
- **分散処理の設計思想:** SparkやMapReduceの根本にある「処理をデータの近くに持って行く」思想をローカルPythonで体験します。
