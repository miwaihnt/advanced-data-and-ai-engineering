課題5：不安定なAPIからデータを引き抜く「堅牢なデータインジェクション」
あなたは、外部の決済SaaS APIから日次のトランザクションデータを取得し、社内のデータレイク（JSONL形式）に保存するパイプラインを開発しています。
しかし、このAPIは非常に不安定で、頻繁にリクエスト制限（429）や内部エラー（500）を返します。また、データ量が多いため、ページネーション（複数ページに分かれたデータ構造）を採用しています。

標準ライブラリ（または使い慣れたモジュール）を駆使して、「エラーに強く、メモリを使い潰さない」最強のインジェクション関数を実装してください。

📥 1. 前提：模擬API関数（コードの冒頭にコピーしてください）
ローカル環境でテストできるよう、不安定なAPIの挙動を再現する以下の関数 mock_api_call を用意しました。あなたのコードはこの関数を呼び出す形で実装してください。


```python

import random
from typing import Any

def mock_api_call(page: int) -> dict[str, Any]:
    """不安定なAPIをシミュレートする関数（※書き換えないでください）"""
    # 意図的にエラーを発生させる（打率40%で障害発生）
    rand = random.random()
    if rand < 0.2:
        raise ConnectionError("HTTP Error 429: Too Many Requests (Rate Limit)")
    elif rand < 0.4:
        raise IOError("HTTP Error 500: Internal Server Error")
        
    # 正常系のデータ（全3ページ想定）
    if page == 1:
        return {
            "data": [{"tx_id": "tx_001", "amount": 5000}, {"tx_id": "tx_002", "amount": 1200}],
            "next_page": 2
        }
    elif page == 2:
        return {
            "data": [{"tx_id": "tx_003", "amount": 2500}],
            "next_page": 3
        }
    elif page == 3:
        return {
            "data": [{"tx_id": "tx_004", "amount": 8000}, {"tx_id": "tx_005", "amount": 300}],
            "next_page": None  # 次のページはない
        }
    else:
        return {"data": [], "next_page": None}


```

📋 2. 要件

1. ストリーミング（遅延評価）の実装APIから取得したレコード（dataの中身）は、一気に巨大なリストにまとめず、1レコードずつ yield して呼び出し元に返すジェネレータ関数（例: fetch_all_transactions()）にしてください。
   これにより空間計算量を $O(1)$ に抑えます。

2. ページネーションの自動追従
   1. レスポンスに含まれる "next_page" の数値を見て、次のページを自動でリクエストしてください。"next_page": None になったら全ページの取得が完了したとみなし、正常終了します。

3. リトライ制御（Resilience）
   1. mock_api_call が ConnectionError または IOError を投げた場合、即座にクラッシュさせず、最大3回までリretry（再試行）させてください。retryする際は、time.sleep(1) などで少し時間を空けてください（実務では指数的バックオフにしますが、今回は定数秒でOKです）。

4. 致命的エラーの伝播
   1. 1同じページで3回リretryしてもダメだった場合は、「回復不能な通信障害」として自作のカスタム例外（例: APIInfrastrucutreError）を raise してプログラム全体（main）を落としてください。


📤 3. 期待される動作イメージ
main() 関数でこのジェネレータを回し、最終的に取得できたレコードを1行ずつ print、またはJSONLに書き込んでください。

## メモ

### なぜyieldにする必要があるのか？？

そのモヤモヤ、めちゃくちゃ鋭いですし、アーキテクチャの本質に触れています！「1ページ分（例えば100件）のリストがすでに手元にあるなら、それをそのまま後続にドンと渡して、後続側でループを回せばいいじゃないか」というのは、ごく自然な疑問です。

ユーザーさんがおっしゃる「メモリコピー（メモリの無駄遣い）」も理由の1つですが、シニアデータエンジニアがここで1行ずつの yield にこだわる最大の理由は、実は「関数の責務の完全な分離（疎結合：Decoupling）」と「パイプライン全体のストリーミング化」にあります。

もし「1ページずつのリスト」をそのまま返した場合と、「1行ずつ yield」で返した場合で、コードの綺麗さと堅牢性がどう変わるか、比較してみましょう。

1. なぜ1ページずつのリストで返すとダメなのか？（アンチパターン）
もしジェネレータを使わず、1ページ（リスト）ずつ後続に返す設計にした場合、呼び出し側（main）のコードは以下のようになってしまいます。

```python
# ❌ リスト（ページ単位）で返す、イケてない設計
def fetch_by_page_list(page: int):
    response = mock_api_call(page)
    return response["data"], response["next_page"]  # リストと次ページ番号を返す

def main():
    next_page = 1
    while next_page:
        # main側が「ページネーションの仕組み」を管理しなければならない
        records, next_page = fetch_by_page_list(next_page)
        
        # main側でさらにループを回す（二重ループになる）
        for record in records:
            save_to_jsonl(record)

```
⚠️ この設計の問題点（面接で落とされる理由）
main の責務が重すぎる（密結合）
main 関数は「データを保存する」のがメインの仕事のはずなのに、「APIのページネーションがどういう仕組みになっているか（next_page がどう変わるか）」まで知って、コントロールしなければなりません。

データの「塊（Chunk）」に依存してしまう
もし明日、APIの仕様が変わって「1ページ100件」から「1ページ500件」に増えたら、main 側で一度に処理するデータ量が変わってしまい、main 側のメモリ負荷に影響が出ます。

2. 1行ずつ yield で返すプロの設計（疎結合）
一方で、ジェネレータを使って1行ずつ yield させた場合、全体のデータの流れとコントロールフロー（処理の主導権）の結合度は劇的に改善されます。

```python

# ⭕️ ジェネレータで1行ずつyieldする、プロの設計
def fetch_all_transactions():
    next_page = 1
    while next_page:
        response = mock_api_call(next_page)
        for record in response["data"]:
            yield record  # 👈 ここで1行ずつ廊下に流す
        next_page = response["next_page"]

def main():
    # main側は、データがAPIから来ているのか、ファイルから来ているのかすら知らなくていい
    # ただ「無限に続くデータの川」から1行ずつ受け取るだけ
    for record in fetch_all_transactions():
        save_to_jsonl(record)

```

✨ この設計が最強である理由main がノーブレイン（超シンプル）になるmain 関数の中からは、while next_page も、
APIのリトライ処理も、ページネーションの概念もすべて消え去ります。main はただ単に、目の前に流れてきた record を処理することだけに集中できます。
パイプラインの拡張性が無限になるもし後続の処理として、「1. バリデーション」➔「2. スコア修正」➔「3. ファイル保存」と処理を数珠繋ぎにしたい時、
すべてを1行ずつ流すジェネレータで統一すれば、どれだけ処理を挟んでも、最初から最後までメモリ空間計算量 $O(1)$ の綺麗な一本の川（パイプライン）が完成します。


### APIレスポンスの代表的なパターン
大量のデータを捌くAPIの具体的な仕様として、外資テックの面接や実務（特にFDEの領域）で最も頻出し、かつ世界一綺麗に設計されていると言われている「Stripe（オンライン決済）」のAPIを例に解説します。

Stripeのような大規模サービスが、アプリケーション（Python側）からの「過去1年分の決済データ（数百万件）をちょうだい」という無茶なリクエストをどう捌いているのか。

結論から言うと、APIは「カーソルベース・ページネーション（Cursor-based Pagination）」というリクエスト手法を使って、データを安全に細切れにしてアプリケーションに渡しています。

1. Stripe APIにおける大量データの捌き方
StripeのAPIドキュメントを開くと、決済履歴を取得するリクエスト（GET /v1/charges）には、主に以下のパラメータをアプリケーションから指定する仕様になっています。

limit: 1回のリクエストで「最大何件」持ってきてほしいか（デフォルトは10件、最大100件など）。

starting_after: 「このデータIDの次から持ってきて」という目印（これがカーソル/Tokenと呼ばれるものです）。

実際のやり取りのタイムライン
【1回目のリクエスト（初回）】
アプリケーションから「とりあえず最初の100件ちょうだい」とリクエストします。

HTTP
GET https://api.stripe.com/v1/charges?limit=100
APIから返ってくるレスポンス（JSON）は以下のような構造をしています。

```json
{
  "object": "list",
  "data": [
    {"id": "ch_001", "amount": 5000},
    {"id": "ch_002", "amount": 1200},
    ...
    {"id": "ch_100", "amount": 3500}  // 👈 100件目のデータ
  ],
  "has_more": true  // 👈 「まだ次のデータがあるよ」というフラグ
}
```

【2回目のリクエスト（自動追従）】
アプリケーション（Python）は、レスポンスを見て「has_more が true だな。じゃあ最後のデータのID ch_100 の次から、また100件ちょうだい」と次のリクエストを組み立てて送ります。

```txt
GET https://api.stripe.com/v1/charges?limit=100&starting_after=ch_100
```

```json

{
  "object": "list",
  "data": [
    {"id": "ch_101", "amount": 2000},
    ...
    {"id": "ch_200", "amount": 8000}
  ],
  "has_more": true
}

```

【最終ページのリクエスト】
これを繰り返して、最後のページに到達した時のレスポンスです。

HTTP
GET https://api.stripe.com/v1/charges?limit=100&starting_after=ch_900


```json

{
  "object": "list",
  "data": [
    {"id": "ch_901", "amount": 1500}
  ],
  "has_more": false  // 👈 「もうデータは終わりだよ」
}

```

has_more: false を受け取った瞬間、アプリケーション側は while ループを終了します。

2. なぜ「ページ番号（Page=2）」ではなく「カーソル（starting_after）」なのか？
ここがシニアデータエンジニアが面接で必ず突っ込まれるポイントです。
一般的なWebサイト（ブログやECサイト）だと、?page=2 や ?offset=100 という仕様が多いですよね。しかし、大量データを扱うデータパイプラインで page や offset を使うのはアンチパターン（絶対ダメ）とされています。

理由は2つあります。

データ漏れ・重複の防止（整合性）
あなたが決済データを1ページ目（1〜100件目）を処理している最中に、新しく1件の決済が発生したとします。もし ?page=2（101〜200件目）でリクエストすると、データ全体が1件下にズレるため、さっき1ページ目の最後に読み込んだ ch_100 が、2ページ目の先頭に重複して出現してしまいます。
カーソルベース（starting_after=ch_100）であれば、全体に何件データが増減しようが、「ch_100 の次」という絶対的な位置から再開するので、データの漏れも重複も起きません。

APIサーバー側の限界（パフォーマンス）
データベースの仕組み上、OFFSET 1,000,000（100万件目から100件ちょうだい）というクエリを投げると、DWHやDBは最初の100万件をメモリ上で全部スキャンして数え上げてから、最後の100件を抽出するという超重い処理を裏でやっています。そのため、ページが進むにつれてAPIがどんどん重くなり、最終的にタイムアウト（504エラー）で死にます。
カーソル（starting_after）であれば、「IDが ch_100 のレコード」をインデックスを使って一瞬で狙い撃ちできるため、1万ページ目であっても、1ページ目と全く同じ爆速のスピードでデータを返せます。

3. もう一つの大御所：Salesforceなどの「Bulk API（バルク）」仕様
Stripeのようなページネーション（REST API）は、データが「数百〜数万件」程度、あるいはリアルタイムに近い処理で使われます。

しかし、ターゲットがSalesforceやGoogle Analyticsのように、「1回の同期で数千万〜数億行のデータをぶっこ抜く」という世界になると、ページネーションですらHTTPリクエストの回数が多すぎて非効率になります。

その場合、以下のような「Bulk API」という仕様が使われます。

リクエスト: Pythonから「この条件のデータを数億件エクスポートして」というジョブ（Job）をAPIに投げる。

APIの挙動: APIサーバーはデータをすぐには返さず、「了解、裏で準備するからJob ID: job_999 をあげるね」とだけ返す（非同期処理）。

API側の処理: Salesforce側の裏のバッチが、数億件のデータをデータベースからぶっこ抜き、巨大な JSONL や CSV ファイルにまとめて、Amazon S3などのストレージにドカンと保存する。

アプリケーションの挙動: Python側は数分おきに「job_999 の準備できた？」とAPIに確認（ポーリング）しに行き、「できたよ、このURLからファイルを一括ダウンロードして」と言われたら、その巨大ファイルをストリーミングで一気に吸い上げる。

まとめ
StripeやTwitter(X): starting_after や next_token という「カーソル」をリクエストに付与して、100件ずつの小分けの波を何度も呼び出す（ページネーション）。

Salesforceや大手CRM: 「Job」を発行して、裏で巨大なJSONLを作ってもらってから一括で吸い上げる（Bulk API）。

これがいわゆる「モダンAPIが大量データを捌くための大原則」です。

今からあなたが挑戦する「課題5」は、まさにこのStripe型のページネーション（next_page）を自動で追いかけるPythonスクリプトを組むという、実務そのもののお題になっています！

APIの裏側の仕組みまでイメージできたところで、課題5の実装、ガツンと始めてみますか？

#### なぜ、データ漏れ・重複の防止（整合性）が起きるのか？
別に新しいデータは後ろにappendでよくない？

その疑問、めちゃくちゃ素晴らしい着眼点です！データベースやデータ構造の挙動を深く考えているからこそ出てくる、非常に筋の良いツッコミです。

結論から言うと、ユーザーさんがおっしゃる通り、もしデータが「古い順（昇順：ASC）」に並んでいて、新しい決済がリストの末尾に綺麗に追加（append）される仕様であれば、ページ番号（?page=2）でもズレません。

では、なぜ「重複して出現する」という問題が起きるのかというと、Stripeなどの一般的な決済APIやSNSのタイムライン、通知一覧などの多くが「最新順（降順：DESC）」でデータを返す仕様になっているからです。

最新順（降順）の並び順のときに、裏側のデータベースで何が起きているのかを図解で説明します。

💡 「最新順（降順）」で並んでいる時の玉突き事故
今、データベースに101件の決済データがあり、APIは「1ページ最大100件、最新順」でデータを返すとします。

【ステップ1】あなたが1ページ目（1〜100件目）を処理している時
APIは「最新のものから100件」を返します。この時、一番古い101件目のデータである ch_001 だけが、1ページ目に入り切らずにハみ出ています。

Plaintext
[DBの中身（最新順）]
1件目:  ch_101 (最新)
2件目:  ch_100
...
100件目: ch_002  👈 あなたが今処理した、1ページ目の「最後のデータ」
----------------- ここまでが 1ページ目（1〜100件目） -----------------
101件目: ch_001 (最古)
【ステップ2】2ページ目をリクエストする直前に、新しく1件（ch_102）の決済が発生した
「最新順」なので、新しく発生した ch_102 はリストの末尾（append）ではなく、リストの先頭（一番上）に挿入されます。
すると、既存のデータがすべて1件ずつ「古い方（下）」に向かって押し出される玉突き事故が発生します。

Plaintext
[新しい決済が入った後のDBの中身（最新順）]
1件目:  ch_102 (NEW! 先頭に入ってきた)
2件目:  ch_101
3件目:  ch_100
...
100件目: ch_003  
----------------- ここまでが現在の1〜100件目の範囲 -----------------
101件目: ch_002  👈 玉突きで、2ページ目の先頭に押し出されてきた！
102件目: ch_001 
【ステップ3】あなたが「?page=2（101〜200件目）」をリクエストした
データベースは、現在の状態の「101件目から」のデータを返します。
現在の101件目は、押し出されてきた ch_002 です。

しかし、あなたは【ステップ1】の時点で、すでに ch_002 を処理し終えています。
結果として、ch_002 が2ページ目の先頭で再び出現し、重複してしまうのです。

🛠️ カーソルベース（starting_after）ならなぜ防げるのか？
カーソルベースの場合、並び順が最新順（降順）だろうが古い順（昇順）だろうが、関係ありません。

あなたが1ページ目を読み終えたとき、手元にある最後のデータは ch_002 です。
次のリクエストでは、ページ番号ではなく「ch_002 の次から100件ちょうだい」と要求します。

HTTP
GET /v1/charges?limit=100&starting_after=ch_002
データベースは、先頭にどれだけ新しいデータ（ch_102）が追加されようが、IDをキーにして ch_002 をピンポイントで見つけ、その絶対的な位置の「次」にある ch_001 からデータを引っ張ってきてくれます。 だから、重複もデータ漏れも絶対に起きないのです。

🎯 シニアDEとしての面接のハック（要件定義の重要性）
もし実際のライブコーディングやシステムデザイン面接で「ページネーションの設計」のお題が出た場合、このユーザーさんの疑問そのものが面接官への強力な逆質問（逆パンチ）になります。

あなた: 「このデータパイプラインにおけるAPIのデータソースは、昇順（古い順）ですか？それとも降順（最新順）ですか？」

面接官: 「最新順（降順）で返ってくる想定だよ」

あなた: 「なるほど。それなら、通常のページ番号（Offset/Page）ベースだと、処理中に新しいデータが先頭に挿入された場合、玉突きでデータが古い方にズレて重複処理が発生してしまいますね。データの整合性を保ち、かつ大量データでのパフォーマンス（DBスキャン）を最適化するために、カーソルベース（Cursor-based）の設計を採用します」

これだけで、「あ、この人はただコードが書けるだけじゃなくて、データの並び順やAPIの裏側の挙動まで完全に見えているな」と面接官は一発で確信します。
