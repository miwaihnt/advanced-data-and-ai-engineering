## 課題2：

O(N)での高速な「インメモリ結合と集計」トランザクションログ（購入履歴）と、ユーザーのマスターデータ（国籍など）を結合し、
「国別の合計購入金額」を算出する関数 aggregate_by_country(transactions, users) を実装してください。

📌 要件計算量の最適化:処理速度を意識し、二重ループ（$O(N^2)$）を使わずに結合すること。

ヒント：マスターデータを事前に辞書（ハッシュマップ）に変換しておくことで、検索を $O(1)$ で行えます。
欠損データのハンドリング:トランザクションに存在するがマスターに存在しない（未知の）ユーザーの購入は、国名を "Unknown" として集計すること。出力形式:{"国名": 合計金額} の形式の辞書（または defaultdict）を返すこと。📥 入力データ

例Pythontransactions = [
    {"user_id": "u1", "amount": 1000},
    {"user_id": "u2", "amount": 2000},
    {"user_id": "u1", "amount": 500},
    {"user_id": "u99", "amount": 3000} # マスターに存在しないユーザー
]

users = [
    {"user_id": "u1", "country": "JP"},
    {"user_id": "u2", "country": "US"},
    {"user_id": "u3", "country": "UK"}
]

📤 期待される出力Python{
    "JP": 1500, 
    "US": 2000, 
    "Unknown": 3000
}

💡 実装のステップ

（おすすめのアプローチ）users のリストをループで回し、{"user_id": "country"} の形をした検索用の辞書を新しく作成します。集計結果を保存するための辞書（collections.defaultdict(int) を使うと便利です）を用意します。transactions のリストをループで回し、1で作った検索用辞書から user_id をキーにして国名を取得（無い場合は "Unknown"）し、金額を加算していきます。準備ができたら、コードを書いて見せてくださいね！レビューと計算量の解説を行います。

## コードに対する補足
3. 実務・実務での「限界値」の語り方このコードは素晴らしいですが、コード査読者からは必ず「じゃあ、ユーザーマスタが10億人（数TB）いたらどうする？」と厳しい設計検証が飛んできます。現在の設計は、マスターデータ（user_master）をすべてメモリ（RAM）に載せる「インメモリ・ハッシュジョイン」という手法です。100万件〜1000万件のユーザーの場合:文字列の短いキーと値のペアであれば、Pythonの辞書で100万件保持しても数百MB程度です。最新のサーバー（メモリ16GB〜32GBなど）なら全く問題なく動きます。このコードで正解です。1億件〜10億件の場合（メモリ枯渇）:辞書のサイズが数十GBに膨れ上がり、Pythonが MemoryError を起こしてクラッシュ（OOM: Out Of Memory）します。実務でここまで聞かれたら、以下のように答えるのがシニアの最適解です。「現在のコードは、時間計算量 $O(N)$、空間計算量 $O(N)$ のインメモリ処理です。数百万件程度なら最速ですが、もしマスタデータがメモリに乗らない数億件のスケールになる場合は、このアーキテクチャは破綻します。その場合は、PolarsのLazy APIを使ってメモリ外（Out-of-core）でJoinさせるか、あるいはマスターデータを事前にRedisやDynamoDBなどのKVS（キーバリューストア）に入れておき、1行ずつAPIでルックアップする設計に変更します。」ご自身のコードが「なぜ速いのか」「どこで限界が来るのか」を正確に言語化できるようになれば、メガベンチャーのコード査読者も唸るレベルです。処理の裏側でメモリがどう動いているかまで想像できたのは、本当に素晴らしいエンジニアの思考プロセスです！

## 標準ライブラリを使うこと
設計のベストプラクティスを伝えるステップシニアロールの実務で最も高く評価されるのは、以下の順序で回答を組み立てるエンジニアです。自力で解く: まず標準ライブラリ（dict）で $O(N)$ のコードをバグなしで素早く書き上げる。自ら限界を指摘する: 「このコードの時間計算量は $O(N)$ で最速ですが、空間計算量も $O(N)$ です。マスタデータが数億件スケールになれば、メタデータのオーバーヘッドによりメモリ（RAM）を数十GB消費してOOM（Out Of Memory）で破綻します」と語る。実務解を提案する: 「したがって実際の本番パイプラインであれば、素のPythonではなくPolarsを採用し、Parquetのメタデータを活かしたPushdownや、ストリーミング処理（Lazy API）でメモリ枯渇を防ぎながら並列処理させます」と締める。この「アルゴリズムの基礎力」と「モダンな実務の解像度」を両方持っている候補者は市場に非常に少ないため、コード査読者に強いインパクトを残せます。
