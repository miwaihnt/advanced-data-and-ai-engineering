課題4：メモリを枯渇させない「ストリーミング処理（遅延評価）」
数GBの巨大なログファイルを処理する想定で、ジェネレータ（yield）を使った関数を実装してください。

要件:

入力は文字列のリスト（巨大なファイルを1行ずつ読み込むイテレータの想定）とする。

行内に "ERROR" という文字列が含まれている行のみを抽出する関数 filter_errors(lines) を作成する。

この関数はリストを返すのではなく、結果を1件ずつ yield すること。

入力例:

Python
log_lines = [
    "[INFO] Server started",
    "[ERROR] DB connection failed",
    "[WARN] High memory usage",
    "[ERROR] Timeout exception"
]
*   **期待される動作:**
    ```python
    # 以下のコードを実行した際、メモリに全件を保持せず正しくループが回ること
    for error_log in filter_errors(log_lines):
        print(error_log) 
    # 出力:
    # [ERROR] DB connection failed
    # [ERROR] Timeout exception

----

## アーキテクチャ議論
「ファイル形式を何にするか」という問いは、データエンジニアリングのアーキテクチャ設計において非常に重要かつ、実務でも頻出のトピックです。

結論から言うと、「データのフェーズ（生データか、分析用か）」によって最適なファイル形式は明確に分かれます。

外資系DEの実務で「なぜそのファイル形式を選んだのですか？」と聞かれた際に、完璧な回答ができるよう、実務で使われる4大フォーマット（CSV, JSONL, Parquet, Avro）の特徴と使い分けを解説します。

1. JSONL (JSON Lines) ＝ 「生データの収集・ストリーミング」の覇者
あなたが前回まで書いていた形式です。1行ずつが独立したJSONになっているものです。

特徴: 行指向（Row-based）、テキスト形式。

メリット:

ネスト（階層）された複雑なデータを持てる。

1行ずつ読み込めるため、前回学んだストリーミング処理（yield）と最高に相性が良い。

テキストエディタで開いて人間が中身を読める。

デメリット: キー名を毎回保持するためファイルサイズが無駄に大きくなる。

ユースケース: アプリケーションの出力ログ、APIからのデータ取得直後（Data Lakeの入口＝Bronze層）。

2. Parquet（パーケット） ＝ 「分析・データウェアハウス」の絶対王者
現在のデータエンジニアリング界隈で最も重要なフォーマットです。ビッグデータ分析（Snowflake, BigQuery, Databricksなど）の裏側はだいたいこれです。

特徴: 列指向（Columnar-based）、バイナリ形式。

メリット:

圧倒的な圧縮率と読み込み速度。

「特定の列（カラム）だけ」を取り出すのが爆速（列指向のため）。

型情報（int, datetimeなど）を厳密に保持できる。

デメリット: バイナリなので人間が直接テキストエディタで読めない。1行ずつ追記していくようなストリーミング処理には不向き。

ユースケース: クレンジングが終わった後の分析用データ（Data Lakeの出口＝Silver/Gold層）。

3. CSV ＝ 「非エンジニアとの連携」のレガシー
特徴: 行指向、テキスト形式。

メリット: Excelで開ける。ビジネスサイドの人が一番扱いやすい。

デメリット: データエンジニアリングの観点では最悪のフォーマット。

型情報が消える（全部ただの文字になる。00123が123になったりする）。

カンマや改行がデータ自体に含まれているとパース（解析）が壊れる。

ユースケース: 最終的なレポートの出力や、ビジネス側からのマスターデータの受領のみ。

4. Avro（アブロ） ＝ 「Kafkaなどリアルタイム通信」のプロ
特徴: 行指向、バイナリ形式。

メリット: スキーマ（データの構造）を厳格に管理でき、かつ行ベースなので「次々と流れてくるデータ」を書き込むのが速い。

ユースケース: Apache Kafkaなどを使ったリアルタイム・ストリーミング処理のシステム間通信。

💡 シニアDEとしての「今のあなた」への推奨
もし現在、Pythonの処理（データパイプライン）のコーディングスキルを鍛える目的で練習しているのであれば、まずは JSONL（または生のテキストログ） を前提に組むことを強くお勧めします。

【理由】

前回学んだ「1行ずつメモリに乗せる（yield）」というストリーミング処理の基礎概念を、標準ライブラリ（jsonやopen）だけで最も直感的に学べるからです。

Parquetはバイナリ形式のため、Python標準の open() では読めず、pandas や pyarrow、DuckDB といった外部ライブラリを使う必要があります。これらは内部でC++などが良しなにメモリ管理をしてくれるため、「自力でメモリをコントロールする力」を養うフェーズでは少しブラックボックスになりすぎます。

実務での「論理的」アピール（実装例）
コード査読者：「今回のパイプライン設計で、入力と出力のファイル形式は何を想定していますか？」

あなた（設計のベストプラクティス）:

「入力側の生データ（Landing Zone）は、スキーマの柔軟性とストリーミング処理のしやすさから JSONL を想定しています。これをPythonのジェネレータを用いてメモリ空間をO(1)に抑えながらクレンジング（ETL）します。
ただし、出力側（Data Warehouseへのロード前）は、後続の分析クエリ（BIツールなど）のパフォーマンスを最適化し、I/Oコストを下げるために、列指向フォーマットである Parquet に変換して保存するアーキテクチャとします。」

これを言われたら、コード査読者は「あ、この人完全に実務のモダンデータスタックを理解してるな」と一発で確信します。

まずは JSONL を読み込んで処理し、最後に出力を JSONL で吐き出すパイプラインを完成させてみましょう！ 次のステップとしてそれを Parquet で出力するように改修していくのが、一番綺麗なロードマップです。

## 🚀 遅延パース（Pre-filtering）によるCPU/メモリ最適化の設計思想

巨大なログファイルをPythonでストリーミング処理する際、単純に「全行をJSONにパースしてから条件判定する」のは、データエンジニアリングの観点で大きな設計ミス（アンチパターン）になります。

### ❌ 非効率なアプローチ（Eager Parsing：即時パース）
```python
for line in f:
    data = json.loads(line)  # すべての行で重いデコード処理が発生
    if "ERROR" in data["log"]:
        yield data["log"]
```

### ⭕ 最適化されたアプローチ（Lazy Parsing / Pre-filtering：遅延パース）
```python
for line in f:
    if "ERROR" in line:  # 生テキストの段階で高速フィルタ
        data = json.loads(line)  # 該当行のみデコード
        yield data["log"]
```

### 🧠 なぜJSONのパース（`json.loads`）は重いのか？

1. **構文解析（Parsing）のオーバーヘッド**
   文字列スキャン（`in` 演算子）は、C言語レベルで最適化された高速なメモリ探索アルゴリズムで動作します。一方、`json.loads()` は文字列を一文字ずつ舐め、文法チェック、ダブルクォーテーションのペア、コロン、エスケープシーケンスなどを解釈する複雑な字句・構文解析を行うため、CPU命令サイクルを大量に消費します。

2. **メモリ確保（Pythonオブジェクトの生成）**
   `json.loads()` が走るたびに、Pythonはヒープ領域に新しい `dict` オブジェクト、キーとなる `str` オブジェクト、値となる `str` オブジェクトを新規確保します。
   - **`in` 演算子**: 追加のオブジェクト生成は **0バイト**。
   - **`json.loads()`**: 辞書と文字列の生成で毎回メモリ割り当てが発生。

3. **ガベージコレクション（GC）の圧力**
   一時的に作成された辞書や文字列オブジェクトは、ループ内の判定が終わった瞬間に不要（ゴミデータ）になります。何百万行ものログを処理する際、数千万個のオブジェクトがGCの対象となり、メモリの回収処理（GCスイープ）が頻繁に割り込むことで、プログラム全体の実行速度が著しく低下します。

#### 📊 1,000万行のデータ処理での負荷比較（エラー率 0.1% の場合）

| 評価指標 | ① Eager Parsing (即時パース) | ② Lazy Parsing (遅延パース) |
| :--- | :--- | :--- |
| **`json.loads()` の実行回数** | **10,000,000 回** | **わずか 10,000 回** (99.9% 削減) |
| **新規オブジェクト生成数** | **30,000,000 個以上** | **わずか 30,000 個** (99.9% 削減) |
| **GCのオーバーヘッド** | 非常に高い (処理のボトルネックになる) | **極めて低い** |

実務および実務においては、「メモリフットプリントを減らすためにジェネレータ（`yield`）を使う」だけでなく、**「CPU負荷とGC負荷を減らすために、デコード等の重い処理は可能な限り後段に遅延させる（Pre-filtering）」**というアプローチを明示することが、シニアエンジニアとしての大きな加点ポイントになります。
