# 💻 課題10：リアルタイム移動窓集計（Sliding Window with Deque）


---

## 📖 【ユースケースとシステム背景】
大規模分散システムにおいて、リアルタイムの異常検知（Anomaly Detection）を行う監視エージェントを設計しています。
システムの各コンポーネントからエラーが発生するたびに、そのエラーが発生したタイムスタンプ（Unix Epochタイム秒単位）が、ミリ秒または秒精度の数値（`float` または `int`）としてリアルタイムでErrorMonitorコンポーネントに流れてきます。

あんたの任務は、新しいエラーが検知されるたびに、**「今この瞬間（エラー検知時刻）から、過去60秒間（`[timestamp - 60.0, timestamp]`）の間に何件のエラーが発生したか」**を即座に計算して返す、スレッドセーフ（非同期またはアモルタイズド $O(1)$）な監視クラス `ErrorMonitor` を実装することです。

---

## 📌 【機能要件とインターフェース】

以下のインターフェースを持つ `ErrorMonitor` クラスを実装しなさい。

### 1. `__init__(self, window_size_seconds: float = 60.0)`
*   インスタンスを初期化し、移動窓（ウィンドウ）のサイズ（デフォルト：60秒）を設定します。

### 2. `record_error(self, timestamp: float) -> int`
*   エラーが発生したタイムスタンプ（`timestamp`）を受け取り、記録します。
*   記録した瞬間に、現在の窓内（`[timestamp - window_size_seconds, timestamp]`）にある有効なエラー件数を返します。
*   **前提条件**: タイムスタンプは常に単調増加（古い順）で流れてくるものとして扱って構いません。

### 3. `get_error_count(self, current_timestamp: float) -> int`
*   新しいエラーを記録することなく、指定された現時刻（`current_timestamp`）において、過去の窓内（`[current_timestamp - window_size_seconds, current_timestamp]`）に残っている有効なエラー件数を返します。
*   これが必要な理由は、エラーがしばらく発生していない静的な期間においても、時間経過によって古いエラーが窓の外へ外れていくため、現在の正確なエラー数をポーリングして取得できるようにするためです。

---

## ⚙️ 【シニアDEとして満たすべき非機能要件（重要）】

1.  **空間計算量の制限（メモリリーク・OOMの絶対回避）**:
    *   システムが数ヶ月・数年間稼働し続けたとしても、蓄積されたすべての過去データをメモリに保持し続けるのは絶対にNGです。
    *   新しいデータが到着した時、あるいは `get_error_count` が呼ばれたタイミングで、窓（Window）の外側へはみ出した古いタイムスタンプは、速やかにメモリから破棄（削除）しなければなりません。
    *   常に、保持するデータ数は最大でも「窓サイズ内（例えば直近60秒）に発生したエラー数 `N`」に制限し、空間計算量を $O(N)$ に抑えなさい。
2.  **時間計算量（アモルタイズド $O(1)$）の達成**:
    *   各エラーの記録・カウント取得が、格納されている全エラーを走査する $O(N)$ ループになっていては、高スループットなストリーミング処理でボトルネックになります。
    *   両端キュー（`collections.deque`）の左端（最も古いデータ）から順番にパージしていくことで、アモルタイズされた（均された）時間計算量 $O(1)$ でデータをパージ・管理しなさい。

## 🛠️ 【テストケース例】

以下のような挙動を正しく処理できることを検証しなさい。

*   **同一時刻の複数エラー**: 同じタイムスタンプ（例: `10.0`）で連続してエラーが発生した場合も正しくカウントされること。
*   **窓外へのパージ**: `10.0` にエラーを記録し、そのあと `75.0`（65秒後）にエラーを記録した場合、`10.0` のデータが即時にパージされ、カウントが `1` になること。
*   **時間経過によるカウント減衰（recordなしのポーリング）**: `10.0` にエラーを記録（カウント1）し、エラーが起きないまま時刻が `71.0` になった時に `get_error_count(71.0)` を呼び出すと、過去のエラーがパージされて `0` が返ってくること。

## 💡 【シニアDEの視点：本番環境での常時起動とシステムデザイン】

実務のシステム設計において、「このErrorMonitorをどう本番で常時稼働させるか？」と検討する際、コードを複雑にするのではなく、**「役割を分離し、システムデザインとして語れること」**がシニアとしての評価を最大化します。

### 1. アプリケーションコードでの常駐タスク化（ポーリングループ）
実務では、メッセージキュー（Apache KafkaやAWS Kinesisなど）から非同期でイベントを受け取る**コンシューマタスク**を無限ループで回し、その中で `record_error` を呼び出します。

```python
async def log_consumer_loop(monitor: ErrorMonitor, queue: asyncio.Queue):
    while True:
        try:
            log_event = await queue.get()
            timestamp = log_event.get("timestamp", time.time())
            
            # コアアルゴリズムの呼び出し
            error_count = monitor.record_error(timestamp)
            
            if error_count >= THRESHOLD:
                await trigger_alert(error_count)
                
            queue.task_done()
        except asyncio.CancelledError:
            break
        except Exception as e:
            logger.error(f"Error in consumer: {e}")
            await asyncio.sleep(1)
```

### 2. 障害復旧（Fault Tolerance）とステート管理
*   **プロセスの死活監視**: コンテナ（Kubernetes）や Linux サービス（`systemd`）で死活監視（自律再起動）を行います。
*   **メモリ上の一時状態（State）の復旧**:
    プロセスがクラッシュして再起動した際、メモリ上の `deque` は空になります。
    プロダクション環境では、**「過去のログ（Kafka等）を60秒前まで巻き戻して再読み込み（Replay）する」**か、**「状態を Redis などのインメモリKVSに同期しておく」**ことで、再起動後も即座に正確な窓集計を再開できる設計にします。

### 3. 分散ストリーム処理エンジン（Apache Flink / Spark）への拡張
もしスループットが極めて高く、単一のPythonプロセスでは処理しきれない場合、自前で無限ループを書くのをやめ、**Apache Flink** や **Spark Structured Streaming** などのマネージドな分散フレームワークに移行します。これらを使えば、ウィンドウ定義（Sliding Window of 60 seconds）を行うだけで、フレームワーク側が自動的にスケールアウト、状態管理、死活監視をすべて面倒見てくれます。
