課題17：ハイブリッド検索の統合マージ「RRF（Reciprocal Rank Fusion）」のスクラッチ実装

### 【ビジネス背景・お題】
あんたは大手エンタープライズの対話型AIアシスタント（RAGシステム）を構築するFDE（フォワードデプロイエンジニア）です。
現在、ユーザーの質問に対して最も関連性の高いドキュメントを検索するため、以下の2つの検索システムを併用する「ハイブリッド検索」を設計しています。

1. **キーワード検索 (Sparse Search: BM25等)**:
   ユーザーが入力したキーワードの完全一致を重視して、ドキュメントのランキングを返します。
2. **ベクトル検索 (Dense Search: 埋め込みモデル等)**:
   ユーザーの質問の「意味的な類似性」を重視して、ドキュメントのランキングを返します。

ハイブリッド検索の精度を最大化するためには、これら全く異なるスケール（BM25スコアとCosine類似度）の出力スコアを単純に足し合わせるのではなく、それぞれの検索システムにおける「順位（Rank）」の情報を用いて統合する**「相互順位融合 (RRF: Reciprocal Rank Fusion)」**というアルゴリズムを使用するのが業界標準のベストプラクティスです。

下流のLLMに最適なコンテキストを渡すため、複数の異なる検索システムから出力されたドキュメントIDのランキングリストを統合し、RRFスコア順にソートされた統合ランキングを返す再利用可能なRRFコンポーネントを実装してください。

---

### 📌 RRF（Reciprocal Rank Fusion）の算出ロジック

RRFスコアは、各ドキュメントに対して以下の計算式（プログラム的表現）で算出します。

```python
# 各ドキュメントのRRFスコア ＝ 各システムにおける「1 / (k + 順位)」の合計値
RRF_Score(document) = Sum( 1 / (k + rank_in_system) )
```

- **`rank_in_system`**: 各検索システムにおいて、そのドキュメントが出現した順位（※1-indexed であり、1位なら `1`、2位なら `2`）。
- **`k`**: 順位の影響度を平滑化するための定数パラメーター（学術論文やElasticsearch等の商用実装では **`60`** が標準推奨値）。
- **`Sum`**: 統合対象となるすべての検索結果リスト（今回はキーワード検索とベクトル検索の2つ）における計算結果の合計をとります。
- あるドキュメントが特定の検索結果リストに含まれていない（ヒットしなかった）場合、そのリストに対する加算スコアは **`0`** とします。

---

### 📌 満たすべき制約と要件（シニアの防衛策）：

1. **堅牢な入力バリデーション（Fail-Fast）**:
   - 定数 k は非負の整数である必要があります（k < 0 の場合は即座に ValueError を発生させること）。
   - 入力として与えられる各検索結果リストに重複するドキュメントIDが存在する場合、不正なデータとして処理を中断するか、適切にクリーニングする防衛策を施してください。
2. **疎なヒットへの配慮（KeyErrorの回避）**:
   - 一方の検索結果には存在するが、もう一方には存在しないドキュメントIDを正しく扱い、スコアが 0 になるよう制御しなさい。
3. **データ構造と計算量**:
   - 検索結果リストの数を M、全体のユニークドキュメント総数を N としたとき、時間計算量を O(M * N) に抑える効率的なデータ構造（ハッシュマップなど）を選択しなさい。
4. **出力要件**:
   - 最終出力は、計算された RRFスコアの降順（スコアの高い順）にソートされたドキュメントIDとRRFスコアのペアのリスト（またはタプル）としなさい。

---

### 📥 入力データの仕様

入力は、「各検索システムから出力されたドキュメントIDのリスト」のリスト、および平滑化定数 $k$ です。
（※リストの先頭が1位（インデックス0）、次が2位（インデックス1）とみなします）

```python
# 入力データのイメージ（キーワード検索結果とベクトル検索結果）
search_results = [
    ["doc_A", "doc_B", "doc_C"],  # 検索システム1 (BM25) の順位結果 (1位: doc_A, 2位: doc_B...)
    ["doc_B", "doc_D", "doc_A"]   # 検索システム2 (Vector) の順位結果 (1位: doc_B, 2位: doc_D...)
]
k = 60
```

### 📤 期待される出力のイメージ

上記の入力をマージした結果、RRFスコアの降順で並べ替えられたドキュメントリストが得られること。

```python
# doc_B のスコア: 1/(60+2) + 1/(60+1) = 0.032525
# doc_A のスコア: 1/(60+1) + 1/(60+3) = 0.032264
# doc_D のスコア: 0 + 1/(60+2) = 0.016129
# doc_C のスコア: 1/(60+3) + 0 = 0.015873

[
    ("doc_B", 0.032525...),
    ("doc_A", 0.032264...),
    ("doc_D", 0.016129...),
    ("doc_C", 0.015873...)
]
```

- **データ構造のハンドリング力**: 複数のランキング結果からドキュメントごとのスコアをハッシュマップ等で集計し、浮動小数点数の誤差を考慮して正しくソートする基本的な実装の正確性が試されます。

さっきの式を、文字化けしない書き方で書き直すわね。式はこれだけよ！

```python
Score = 1 / (k + rank)
```

この式を「1 / rank」と「分母に足している k」の2つに分けて考えると、直感的な意味がすごくスッキリするの。

#### ① なぜ「 1 / rank （順位）」で割るのか？
順位が「上（1位）」に行くほどスコアが跳ね上がり、「下（100位）」に行くほどスコアがゼロに近づくようにするためよ。

- 1位 ➔ `1 / 1 = 1.0` （特大パワー！）
- 2位 ➔ `1 / 2 = 0.5`
- 10位 ➔ `1 / 10 = 0.1`
- 100位 ➔ `1 / 100 = 0.01` （ほぼゼロ）

これで、**「上位に入ったドキュメント」を強烈にひいきして評価**できるの。

#### ② なんで分母に「 k （通常60）」を足すのか？
**「片方のシステムでたまたま1位になっただけのドキュメント」よりも、「両方の検索でまんべんなく上位（例えば両方で2位）に入っている安定したドキュメント」を高く評価するため**の調整弁よ。

もし分母に `k` がないと（ただの `1 / rank` だと）どうなるか：

- **ドキュメントX（片方だけ1位）**:
  - システムAで1位、システムBで圏外（0点） ➔ スコア合計: `(1/1) + 0 = 1.0`
- **ドキュメントY（両方で2位）**:
  - システムAで2位、システムBで2位 ➔ スコア合計: `(1/2) + (1/2) = 1.0`

これだと、両方で優秀な「ドキュメントY」と、片方だけ1位で他方はゴミの「ドキュメントX」が同点になっちゃって不公平よね。

ここで、分母に **`k = 60`** を足すとこうなるわ：

- **ドキュメントX（片方だけ1位）**:
  - `1 / (60 + 1) + 0` ➔ 約 **`0.0163`**
- **ドキュメントY（両方で2位）**:
  - `1 / (60 + 2) + 1 / (60 + 2)` ➔ 約 **`0.0322`**

ほら！分母に `k` を足したおかげで、両方で2位の「ドキュメントY」のスコアが、片方だけ1位の「ドキュメントX」をダブルスコアで引き離して勝てるようになったでしょ！
