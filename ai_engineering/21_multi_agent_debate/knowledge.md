# 📝 課題21 ナレッジ：AIシステム設計（Ollama vs Hugging Face 生ロード）

ローカル環境（特にApple Silicon Mac）でオープンソースLLMを動作させて実験や検証を行う際、Pythonの `transformers` ライブラリで直接ロードするよりも `Ollama` を用いるべき理由。

| 比較項目 | Hugging Face (`transformers` 生ロード) | Ollama (推論サーバー経由) |
| :--- | :--- | :--- |
| **推論速度** | 遅い（PyTorch上でのMetal/GPU最適化が難しく、CPUに落ちやすい） | **極めて高速**（バックエンドの `llama.cpp` がC++でMacのGPU向けに極限まで最適化されている） |
| **メモリ消費** | 巨大（ unquantized モデルは3Bでも6〜8GB、8Bで16GBのRAMを消費） | **省メモリ**（自動的に4-bit量子化等のGGUFモデルが使われ、容量が4分の1に軽量化される） |
| **並行処理** | GIL（グローバルインタプリタロック）やメモリ制限により、Pythonプロセスがクラッシュしやすい | **極めて堅牢**（LLM推論自体はPython外のC++プロセスで処理され、並行キューイングやバッチングも自動で捌く） |
| **API互換性** | 独自の入力・テンソル整形コードを自前で実装する必要がある | **OpenAI互換のローカルREST API** (`http://localhost:11434/v1`) が自動で立つため、エコシステムがそのまま使える |

## 📊 ローカルLLMのモデルサイズ基準 (Qwen2.5:3b)

ローカル環境での実験用に採用する `qwen2.5:3b` モデルのスペックと意義。

- **パラメータ数**: **3B (3 Billion = 30億パラメータ)**。
- **ディスク容量**: **約 1.9 GB**。Ollamaが4-bit量子化（Q4_K_M）という圧縮を自動で行っているため、容量が非常にコンパクト。
- **必要メモリ（RAM）**: **4 GB 以上**。Macのメモリが8GBや16GBあれば、VS Codeやブラウザを開きっぱなしでも、スワップを起こさず裏で軽快に動作する。
- **推論速度（スピード）**: **秒間 50〜75 トークン**（Apple Silicon Macの場合）。非同期の討論ループを何回回してもレイテンシにイライラしない速度。
- **推論能力**: 
  Alibabaが開発した `Qwen2.5-3B` は、「超軽量なのに、少し前の7B（70億）や13B（130億）のモデルに匹敵する、極めて高い数学的推論（GSM8K）とプログラミング能力」を持っている。また日本語の言語表現の追従性が非常に高い。
- **選定の結論**:
  0.5B/1.5B のような軽量すぎるモデルでは「フォーマット崩れ」や「論理エラー」が頻出し、7B/8B では動作が重くなるため、**3Bは「フォーマット遵守力」と「動作の軽さ」を両立するローカル実験のベストスイートスポット**である。
