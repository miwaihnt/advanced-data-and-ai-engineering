# RAG Primitive Benchmark Report (9.1万件規模)

本レポートは、約 9.1 万件の国会会議録データを用い、検索アルゴリズム（インデックス、ハイブリッド検索、リランキング）の性能と精度を定量・定性の両面から評価したものである。

## 1. 実行環境 (Hardware & Environment)
- **OS**: macOS (Sequoia 15.x)
- **CPU/GPU**: Apple M4 (MacBook Pro)
- **RAM**: 16 GB
- **DB**: LanceDB (v0.x.x)
- **Embedding Model**: `intfloat/multilingual-e5-small` (384 dim)
- **Reranker Model**: `hotchpotch/japanese-reranker-cross-encoder-xsmall-v1`
- **Compute Resource**: Apple Silicon **MPS (Metal Performance Shaders)** による GPU 加速

## 2. データセット (Dataset)
- **ソース**: 国会会議録 API (2023年通年データの一部)
- **総チャンク数**: **90,940 チャンク**
- **1チャンクの設定**: 最大 2,000 文字 / オーバーラップ 200 文字
- **総文字数**: 推定 1.5 億 〜 1.8 億文字相当

#### ■ 分析と考察
- **驚異の大逆転劇**: 純粋なベクトル検索では **291 位** に埋もれていた芳賀議員のチャンクが、ハイブリッド検索によって **1 位** に浮上した。これは「具体的な対策（給料増など）」というキーワードを FTS が捉え、リランカーがその内容の深さを正しく評価した結果である。
- **ベクトル検索の限界（Flat #1 の脱落）**: 真のベクトル検索で 1 位だった岸田総理のチャンクは、抽象的な「決意」に終始しており、具体的対策を求めるクエリに対してはリランカーによって「回答としての優先度」を下げられた。
- **結論**: 「ベクトルの近さ」はあくまで「文脈の近さ」に過ぎない。2-Stage Retrieval を導入することで、数百位に沈んでいる「真の正解」を救い出せることが実証された。

## 5. 技術的考察と知見 (Insights)

### 5.1. 「0.1 秒の壁」とインデックスの正当性
データ量が 9 万件を超えたことで、Flat 検索のレイテンシ (0.15s) がユーザーの体感速度を損なうラインに達した。HNSW 導入による約 8 倍の高速化は、この規模において不可欠なトレードオフである。

### 5.2. セマンティック・ハルシネーションの回避
ベクトル検索単体では、クエリの「雰囲気」に流されて無関係なトピックを 1 位にする挙動が確認された。FTS とリランカーを組み合わせる 2-Stage Retrieval は、これを抑制する極めて強力な手段である。

### 5.3. 日本語トークナイズの完全制御
SudachiPy で事前に分かち書きすることで、複合語の検索漏れを防ぎ、ハイブリッド検索の「精度の土台」を盤石にしている。
