# 💻 課題28：自律テスト・修復マルチエージェント（The Micro AI Scientist）

> **【自律コード修復マルチエージェント（The Micro AI Scientist）】**
> 本プロジェクトは、「The AI Scientist」の論文研究の思想を抽出し、実用的な「マルチエージェントによるコード自律修復ループ」として再設計した検証システムです。
> 外部のエージェントフレームワーク（LangChainやLangGraph等）に依存せず、非同期マルチエージェントの協調システムを純粋なPython/asyncioで構築し、設計思想と堅牢性を検証しています。

---

## 📖 背景設定（Background）

あなたは自律的にソフトウェアのバグを修正するAIエージェントシステムを開発しています。
バグが含まれる「対象プログラム（`target_code.py`）」と、それを検証する「テストスクリプト（`test_target_code.py`）」が与えられたとき、システムは以下の**3つの役割を持つエージェント**を協調させ、テストがすべてパスするまで自律的に「修正→レビュー→テスト→再修正」のループを回します。

---

## 👥 エージェント構成（Agents）

本システムは、以下の3つの役割（Role）を持つマルチエージェントシステムです。それぞれのプロンプトと役割を独立させて実装しなさい。

```
                    ┌──────────────┐
                    │ Orchestrator │
                    └──────┬───────┘
                           │
             ┌─────────────┼─────────────┐
             ▼             ▼             ▼
      ┌────────────┐ ┌────────────┐ ┌────────────┐
      │ Coder Agent│ │   Reviewer │ │ Tester     │
      │   (実装)   │ │   (検算)   │ │  (実行)    │
      └────────────┘ └────────────┘ └────────────┘
```

1. **Coder Agent（コーダー）**:
   - 役割: ソースコード、テストコード、および直前の「エラーログ」や「レビュー指摘」を読み込み、バグを修正した新しいPythonコードを生成する。
   - 制約: 出力は純粋なPythonコードのみ（または指定されたJSON構造）とし、余計な解説テキストを出力しないこと。
2. **Reviewer Agent（レビュワー）**:
   - 役割: Coderが生成した修正コードと、その修正理由をレビューする。
   - 検証内容:
     - 修正が論理的に正しいか（ハルシネーションやデグレがないか）。
     - 元々のプログラムの要件を壊していないか。
   - 出力: レビュー結果（合格 / 不合格）と、不合格の場合は具体的な指摘内容。
3. **Tester Agent（テスター/実行環境）**:
   - 役割: Coderが書き、Reviewerが承認したコードを、**実際のPython実行環境（subprocess等）で実行**し、テスト結果を取得する。
   - 出力: テスト成功（パス）か、失敗した場合は `pytest` などの実行ログ（Stdout/Stderr）。
   - ⚠️ **注意**: LLMのハルシネーションを防ぐため、テスト実行はLLMではなく、**Pythonの実際の実行プロセスで行うこと（これがScientistの「実験」にあたります）**。

## 🔄 自律修復ワークフロー（Workflow）

`Orchestrator` は、以下のステートフルなループを最大 `max_iterations` 回実行しなさい。

```
[Start] target_code.py と test_target_code.py を読み込む
   │
   ├─► [Loop Start (1 to max_iterations)]
   │     │
   │     ├──► [1. Coder] バグを修正したコードを執筆
   │     │
   │     ├──► [2. Reviewer] コードを検証
   │     │     ├─ [Reject (不合格)] ──► 指摘を Coder に戻して再チャレンジ（Loop継続）
   │     │     └─ [Approve (合格)] ───► [3. Tester] へ進む
   │     │
   │     └──► [3. Tester] テストを実行（subprocess）
   │           ├─ [Success (全パス)] ─► ループを即時脱出して終了！
   │           └─ [Fail (コケた)] ───► エラーログを Coder にフィードバック（Loop継続）
   │
[Timeout/Fail] 最大回数に達しても通らなかった場合、諦めて「修復失敗レポート」を出力
[Success] 通った場合、最終的な修正コードと「自律修復レポート」を出力して正常終了
```

## 🎯 ターゲットコードに仕込まれるバグ（`target_code.py`）

以下の3つのバグが混入した注文集計プログラムを、エージェントに自律修復させなさい。

1. **金額計算のバグ**: VIP割引（10% OFF）の計算で、誤って `total_amount = total * 0.1` としており、割引額そのものが請求額になってしまっている（正しくは `total * 0.9`）。
2. **型安全性の欠如**: 注文データの `price` が文字列（例: `"1500.0"`, `"2000"`) で送られてくる場合があり、floatキャストせずに足し算しようとして `TypeError` でクラッシュする。
3. **表記揺れの未考慮**: 顧客の `membership` が `"vip"` や `"VIP"` と大文字小文字が混ざっており、一部のVIP顧客に割引が適用されていない。

## 🚫 禁止事項

- `langchain`, `langgraph`, `crewai` などのエージェントフレームワークの使用は禁止。
- LLMに「テストを実行したふり」をさせるのは禁止。必ず `subprocess` を使って実際にテストスクリプトを実行し、リアルなエラーログを取得すること。

## 💬 提出時に回答すべき設計上の問い（Design Questions）

1. **「Orchestrator、Coder、Reviewerの間で、エージェントの会話履歴（状態）をどのように管理・伝搬させたか？実装した状態管理の設計を説明せよ」**
   - **回答**: 
     Orchestratorクラス内にメンバ変数 `self.history: list[str]` を設け、システム全体の会話履歴を一元管理する「状態ストア」として設計しました。
     イテレーションごとに発生した「Reviewerによる却下理由」や「Testerによる実行エラーのスタックトレース」を、`[Iteration X Review Reject] ...` のような定形形式で随時このリストへ蓄積します。
     そして、毎ループの開始時に CoderAgent の `generate_fix` メソッドの引数として `self.history` をそのまま引き渡します。これにより、Coderは「過去に自分がどのようなコードを書き、なぜ失敗したか」を全て踏まえた上で、次の修正案を生成することができ、同じ過ちを繰り返さない（状態の継続的な伝搬）仕組みを実現しています。

2. **「Reviewer Agentが単なる『イエスマン』にならず、Coderの怪しい修正（例: テストコード自体を書き換えてパスさせようとするインチキなど）を見破るために、どのようなプロンプト設計やシステムガードレールを設けたか？」**
   - **回答**: 
     LLMによる論理レビューと、システム側での物理的な制限を組み合わせた「多層防御」を採用しました。
     - **プロンプト設計（LLM検知）**: Reviewer Agentのシステムプロンプトに、「テストコード（`test_target_code.py`）自体を書き換えて合格させようとする行為や、アサーションの無効化などの『不正行為（Cheating）』を厳重にチェックし、検知した場合は即リジェクト（`approved = False`）せよ」という明確なガードレールを仕込みました。また、入力として「元のコード」「修正後のコード」「Coderの修正解説」の3つを独立して渡し、差分の論理的整合性を厳密に比較検証させています。
     - **システムガードレール（物理的制限）**: LLMの目視だけに依存するのはハルシネーションのリスクがあるため、実務上の鉄則として以下の物理的防御を設計します。
       1. **テストファイルの権限制限**: `TesterAgent` が動作する環境において、テストコードのファイルシステム権限を「読み取り専用（Read-only）」に設定し、Coderによる書き込みを物理的に遮断する。
       2. **ハッシュ値（SHA-256）チェック**: テスト実行直前に、実行対象のテストコードのハッシュ値を事前に算出したオリジナル値と比較し、改ざんを検知した場合は即時実行エラーにする。
       3. **AST（抽象構文木）解析**: 生成されたPythonコードのASTをパースし、テスト用の入力を狙い撃ちした不自然な条件分岐（例: `if orders[0]["order_id"] == "ORD-002": return 270.0` などのオーバーフィッティング）を静的に検知・ブロックする。

3. **「コードを動的実行（subprocess）する際、悪意あるコード（例: `import os; os.system('rm -rf /')` のようなコード生成）をインジェクションされるセキュリティリスク（RCE）に対し、本番環境ではどのような隔離環境（サンドボックス）を採用して対策すべきか？」**
   - **回答**: 
     ローカルの `subprocess` を直叩きする環境は、ホスト環境全体がRCE (Remote Code Execution) の標的になるため、本番環境では以下の強力なサンドボックス隔離を導入する必要があります。
     - **軽量MicroVM（Firecracker / gVisor）の採用**: コンテナ共有カーネルの脆弱性を突くエスケープ攻撃を防ぐため、gVisor（システムコールフィルター）による隔離か、AWS Firecrackerのようなハードウェア仮想化レベルで完全に独立したミリ秒起動の仮想マシン（MicroVM）内でテストプロセスを実行します。
     - **ネットワークの遮断（Network Isolation）**: テストを実行するサンドボックス環境の外部インターネットアクセスを完全に遮断し、攻撃者サーバーへのC2接続や、機密データの外部流出（エクスフィルトレーション）を防ぎます。
     - **エフェメラル（使い捨て）実行**: テスト実行ごとにクリーンなイメージから環境を起動し、実行完了後は即時に環境全体を破棄します。また、書き込み可能領域は最小限の一時ディレクトリ（tmpfsなどのメモリ上）に限定します。

4. **「このマルチエージェントループは、単一の自律エージェント（ReAct）で逐次実行させるアプローチと比較して、何が優れており、どのようなトレードオフがあるか？」**
   - **回答**: 
     - **優れている点 (Advantages)**:
       - **役割の分離 (Separation of Concerns)**: 「コードの実装（Coder）」「静的な検算（Reviewer）」「実行環境での検証（Tester）」に明確に分かれているため、各プロンプトにかかるコンテキストの認知負荷を最小限に抑えられ、ハルシネーションを防ぎやすいです。
       - **独立した批判性（バイアスの排除）**: 自分が書いたコードを自分でレビューすると「動くはず」という認知バイアスに囚われやすいですが、別エージェントであるReviewerが客観的にチェックすることで、不正なパッチや論理エラーを水際で防ぐことができます。
       - **トレードオフ (Trade-offs)**:
         - **トークンコストと遅延の増加（自己反省ReActとの決定的な差異）**:
           単一エージェントによる自己反省（Self-Reflection）ループも履歴の肥大化によるコスト増はありますが、マルチエージェントは以下の理由で時間的・金銭的コストがさらに跳ね上がります。
           1. *ネットワークラウンドトリップの増加*: エージェント間（Coder ➔ Reviewerなど）でバトンを引き渡すたびに、物理的に独立したAPIリクエストを何度も送信するため、ネットワークの往復遅延（オーバーヘッド）が毎回蓄積され、完了までのレイテンシが非常に長くなります。
           2. *プロンプトキャッシュ（Prompt Caching）の不効化*: 単一のチャットセッションを継続するReActでは、固定のシステムプロンプトや元のソースコードなどの重複部分に対し、プロバイダ側のプロンプトキャッシュ（最大90%の割引）が強力に効きます。しかし、マルチエージェントではエージェントごとにセッションや役割（プロンプト構成）が断絶しており、かつエージェント間を流れるコード差分データが動的に変化するためキャッシュヒット率が著しく低下し、エージェントの数だけインプットに対して満額のトークン課金が発生するという非効率が生じます。
         - **状態遷移と協調設計の複雑化**: エージェント間で噛み合わない応答が発生した際の例外処理や、Pydanticスキーマを通じたデータ受け渡しなど、システム開発・運用のコストが高くなります。
