# TechTrendWatcher Test Strategy 🛡️

このドキュメントは、Martin Fowler の「Practical Test Pyramid」に基づき、TechTrendWatcher の品質を保証するためのテスト方針を定めたものである。バカ犬でも迷わないように、厳格なルールを敷いてあげたわ！

## 🎯 テストの目的
1. **自信 (Confidence)**: コードを変更しても、既存の機能が壊れていないと断言できること。
2. **速度 (Speed)**: 開発サイクルを回すために、大半のテストが数秒以内に終わること。
3. **投資対効果 (ROI)**: 些細なコード（Getter/Setter）は無視し、バグが潜みやすく、壊れたら痛い「価値のあるロジック」を徹底的に守ること。

---

## 🏗️ テストピラミッドの構成

### 1. Unit Tests (ピラミッドの土台: 80%)
最も高速で、最も数が多いテスト。ネットワーク接続は一切禁止よ！

- **対象**:
    - `github/processor.py`: スターの差分計算 (`star_delta`)、DataFrame の変換ロジック。
    - `models/github.py`: Pydantic によるバリデーションと `raw_data` の保持。
    - `core/exceptions.py`: 例外の階層構造。
- **手法**:
    - **完全なモック化**: `httpx` や外部 SDK は使わず、偽のデータ（Fake Data）を流し込んでロジックの正しさを検証する。
    - **境界値の徹底**: データが 0 件、1 件、最大件数の場合や、予期せぬ型が混じった場合を `pytest.mark.parametrize` で網羅する。

### 2. Integration Tests (ピラミッドの中間: 15%)
「外の世界（API/DB）」との境界線（契約）を守るためのテスト。

- **対象**:
    - `GithubClient` / `NotionClient` / `SnowflakeClient` の API 連携部分。
- **手法**:
    - **狭義の統合テスト**: 本物のネットワークには繋がず、`respx` 等を使って「API が 401 や 429 を返した」という**状況を偽造**し、クライアントが正しく例外を投げるか、リトライするかを検証する。
    - **契約テスト**: API のレスポンス形式が変わっていないか、モックの定義と実際の挙動を照らし合わせる。

### 3. End-to-End (E2E) Tests (ピラミッドの頂点: 5%)
システム全体が繋がって動くかを確認する、重くて壊れやすいテスト。

- **対象**:
    - `main.py` の実行フロー全体。
- **手法**:
    - **サンドボックス環境**: 本番の Notion/Snowflake ではなく、**テスト専用の DB/スキーマ** に対して実際にデータを流し込む。
    - **最小限のケース**: 「GitHub から取得し、差分を検知し、Notion と Snowflake に届く」というハッピーパス（正常系）の 1 ケースだけに絞る。

---

## 🚫 テストしないもの (Trivial Code)
以下のテストに時間を溶かすのは、三流エンジニアのやることよ。

- 単純な `__init__` での変数代入。
- ログ出力 (`logger.info`) の呼び出し確認。
- Pydantic や Polars などのサードパーティライブラリ自体の基本機能。
- 定義ファイル (`config.py`) の読み込み（バリデーションロジックがない場合）。

---

## 🛠️ 使用ツール & 規約

- **Test Runner**: `pytest`
- **Async Support**: `pytest-asyncio`
- **Mocking**: `pytest-mock` (mocker), `respx` (for httpx)
- **Coverage**: `pytest-cov` (目標カバレッジ: 価値のあるロジックに対して 100%)
- **Isolation**: `.env.test` を使用し、テスト実行時は必ず環境を切り替えること。

---

## 🤨 守るべき鉄則
- **「ネットワーク不要」をデフォルトに**: `pytest` を実行するのに Wi-Fi も API トークンも不要な状態を維持しなさい！
- **「独立性」を保て**: テスト A の結果が テスト B に影響するなんて、絶対にあってはならないわ！
- **「壊れたら叫ぶ」**: テストが落ちたら、原因が一目でわかるようなエラーメッセージを（モックの定義等に）仕込みなさい。
