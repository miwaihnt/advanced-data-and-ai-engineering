# TechTrendWatcher: Architectural Blueprint & Review Guide 🧠

このドキュメントは、バカ犬（ユーザー）が「何も見ずにこのシステムを再構築できる」ようになるための、設計思想と復習ポイントをまとめたものである。Tier 1 企業の面接官を黙らせるための「武器」にしなさい！

---

## 🏗️ 1. アーキテクチャ設計のコア（The "Why"）

設計を始めるとき、まずこの 3 つの問いに答えられなければならないわ。

### なぜ非同期 (asyncio) なのか？
- **理由**: このアプリは「GitHub API」「Notion API」「Snowflake API」という 3 つの外部 I/O が主役。
- **メリット**: I/O 待ち時間を有効活用して、複数のクエリや複数の Notion 書き込みを並列で実行できる。同期処理（一つずつ順番に待つ）に比べて、スループットが数倍〜数十倍に跳ね上がるわ。

### なぜ Pydantic なのか？
- **理由**: API から来るデータは「未加工の荒野」よ。
- **メリット**: 型の保証（Type Safety）だけでなく、`model_validator` を使って「生データを VARIANT 用に保持する」という高度なデータ処理を、モデルの層で完結させられるの。

### なぜ Polars なのか？
- **理由**: データエンジニアなら速度とメモリ効率に拘（こだわ）りなさい。
- **メリット**: Pandas よりも高速で、メソッドチェーンによる「宣言的なデータ加工」が可能。特に Parquet との相性が抜群よ。

## 🏎️ 2. データフローの構築（The "Flow"）

パイプラインを組むときは、常に **「ETL (Extract, Transform, Load)」** を意識しなさい。

1.  **Extract (取得)**: 
    - `GithubClient` が API から JSON を引っこ抜く。
    - ここでのポイントは **「リトライ戦略（tenacity）」**。一時的な不機嫌（429/500）で止まらないレジリエンスを組み込むこと。
2.  **Transform (加工)**:
    - `Processor` が Polars を使ってスターの差分を計算する。
    - **「前回データとの Join」** が鍵。ディスク（Parquet）とメモリ（current_df）をどう繋ぐか、そのロジックを完璧に説明できるようにしなさい。
3.  **Load (ロード)**:
    - `NotionClient` (Upsertロジック) と `SnowflakeClient` (バルクロード)。
    - **「冪等性（Idempotency）」** を意識しなさい。2 回実行しても、Notion に同じページが 2 つ作られないように設計したわよね？

## 🛡️ 3. Engineering Excellence（プロのこだわり）

ただ動くだけなら素人。プロはここが違うのよ！

- **Observability (可観測性)**: 
    - `structlog` による JSON ログ。
    - メッセージの中にデータを埋め込まない。「イベント名」と「コンテキスト（キー）」を分ける。これができないなら Tier 1 の面接には行かないことね。
- **Error Handling (例外設計)**:
    - ライブラリの例外をそのまま投げない。
    - `ExternalAPIError` ➡ `SnowflakeAPIError` みたいに、自分たちのドメイン（意味）に合わせて包み直す（Wrap）。
    - Snowflake の `errno` みたいに、詳細な理由（reason）まで特定してログに出す。
- **Test Strategy (テストピラミッド)**:
    - **Unit**: ロジック（スター計算）を 100% 保証。
    - **Integration**: `respx` で外の世界を騙し、異常系（429/500/Timeout）をすべて網羅。
    - **Mocking**: `asyncio.sleep` をモックしてリトライテストを爆速にする。この「ハック」を知ってるのは、あんたみたいな幸せなバカ犬だけよ！
